### מבוא כללי

### מבוא לאוטומציה ואינטגרציה — https://help.sumit.co.il/he/articles/11582017

### טריגרים (8)

### מבוא לטריגרים — https://help.sumit.co.il/he/articles/6324125
- מודול טריגרים: פעולות אוטומטיות (מסר/מסרון/עדכון סטטוס) וקריאות Webhook בפורמט JSON על יצירה/עדכון כרטיסים.
- **מנגנון retry של webhook:** המתנה 10 שניות לתגובת HTTP 200 → ניסיון חוזר אחרי 30 שניות → סה"כ **5 ניסיונות**, ואז **השהיית הטריגר**.
- המלצה: השרת המקבל יחזיר 200 מיידית ויבצע עיבוד אסינכרוני.
- מגבלה: עדכון כרטיסים דרך טריגר פנימי מוגבל לשדות טקסט בתיקייה הרלוונטית בלבד.

### אינטגרציה מלאה עם Make — https://help.sumit.co.il/he/articles/5840257
- אפליקציית SUMIT רשמית בתוך Make; חיבור ב-3 שלבים; תרחישים עם Triggers ו-Actions של SUMIT; פרמטרים קבועים או דינמיים מהטריגר. לא צוינו מגבלות/מכסות.

### אינטגרציה מלאה עם Zapier — https://help.sumit.co.il/he/articles/5840259
- אפליקציית SUMIT ב-Zapier ללא צורך בקוד; חיבור ב-3 שלבים; טריגרים לפי תיקייה ותצוגה; Actions עם פרמטרים קבועים/דינמיים.

### תצוגה מוכנה לבניית אוטומציה אחרי תשלום — https://help.sumit.co.il/he/articles/5846658
- תצוגת "נתונים לטריגר" במודול סליקות אשראי: רק סליקות מוצלחות, כולל מייל+טלפון של הלקוח.
- עובדת על **כל תשלום** במערכת (לא רק דפי תשלום); האוטומציה עצמה דרך Zapier/Make.

### כיצד יוצרים טריגר שישלח Webhook ל-Make — https://help.sumit.co.il/he/articles/10442304
- 5 שלבים: התקנת מודולים → תצוגה מותאמת עם פילטרים → קבלת כתובת Webhook מ-Make → יצירת טריגר → הגדרת קריאת HTTP מסוג JSON.
- ה-webhook שולח את כל שדות הכרטיס שעודכן ב-JSON.
- **מגבלה:** שדות מקושרים מתיקיות אחרות (מייל/טלפון מכרטיס הלקוח) לא נשלחים — נדרשת קריאת API נוספת לפי ה-ID שהתקבל.

### יצירת התראה אוטומטית למייל בעקבות פעולה במערכת — https://help.sumit.co.il/he/articles/11476911
- תבנית: שדה מייל קבוע בתיקייה → תצוגה → תבנית מסר במודול דיוור → טריגר.
- עובד על כרטיסים שנוצרו ידנית, דרך דף תשלום או דרך API.

### יצירת טריגר — https://help.sumit.co.il/he/articles/11477015
- שדות הטריגר: תיאור, תיקייה, תצוגה, סוג השינוי המפעיל, שלבי ביצוע, סטטוס, קישור למסר (מייל/SMS/webhook).

### שליחת Webhook ממערכת סאמיט — https://help.sumit.co.il/he/articles/11577644
- Webhook = HTTP request לכתובת חיצונית בפורמט JSON או FORM; מכיל את כל שדות הכרטיס כפי שמוצגים בתצוגה שנבחרה.
- אין במאמר פירוט אימות/חתימה על ה-payload (אין מנגנון signature מתועד).

### אינטגרציה עם סליקת אשראי (5)

### Payments JavaScript API — https://help.sumit.co.il/he/articles/5893615
- ספריית JS להטמעת דף תשלום מעוצב עצמאי שקולט פרטי אשראי בצורה מאובטחת דרך SUMIT — מצמצם את דרישות תקן PCI על העסק.
- Tokenization: נוצר שדה `og-token` (טוקן חד-פעמי) שנשלח לשרת שלך.
- זרימה: טעינת הספרייה → אתחול עם פרטי העסק → טופס HTML → חיוב בצד השרת דרך Charge/Transaction API.

### כמעט כל האינטגרציות שאפשר לעשות עם SUMIT — https://help.sumit.co.il/he/articles/5832819
- 4 מסלולי אינטגרציה: Payments JS (דורש מתכנת בסיסי), תוסף WordPress/WooCommerce, Zapier/Make, API מלא.
- endpoints מרכזיים: `/billing/payments/beginredirect/` (דף תשלום מאובטח), `/billing/paymentmethods/setforcustomer/` (טוקניזציה לחיובים עתידיים), `/billing/payments/charge/` (חיוב חד-פעמי), `/billing/recurring/charge/` (הוראת קבע).
- חשבונית נוצרת אוטומטית בסליקה; ניתן להגדיר יצירת טיוטות בלבד.

### לסלוק אשראי עם API — https://help.sumit.co.il/he/articles/5833033
- REST API לסליקה ישירה באתר ללא הפניה לדף חיצוני; SUMIT ברמת PCI DSS Level 1, אך העסק חייב לעמוד בדרישות אבטחה אם הוא נוגע בפרטי כרטיס.
- הוראות קבע: המערכת מחייבת אוטומטית לפי תדירות הגבייה שהוגדרה.
- **פרט טכני:** בקריאת `billing/recurring/charge` יש להגדיר `UpdateCustomerByEmail=True` כדי שמסמכים יישלחו ללקוח לאחר חיובים.

### אינטגרציה עם סליקת אשראי — https://help.sumit.co.il/he/articles/5832880
- מודולים נדרשים: הכנסות + סליקת אשראי (+ הוראות קבע לחיובים חוזרים).
- 3 מסלולים: API לסליקה חד-פעמית עם מסמכים אוטומטיים, API להוראות קבע עם מעקב אוטומטי, דף Redirect מאובטח.
- המערכת מטפלת בשליחת מסמכים בדוא"ל ובתזכורות עדכון כרטיסים שפג תוקפם.
- מוצרים/שירותים אפשר להגדיר מראש או להעביר בזמן החיוב בקריאת ה-API.

### שימוש בכרטיסי בדיקות לבדיקת אינטגרציה עם סליקת אשראי — https://help.sumit.co.il/he/articles/5832877

## חוקים ומגבלות מרכזיים

### מנגנון ActionsBilling / obligo — שלושה מנגנונים נפרדים, לא לערבב

**1. Rate limit טכני (throttle) — לפי דקה, זמני.**
- מקור: "מגבלות ניהול המידע במערכת" (9394082): **עד ~100 קריאות API בדקה; חריגה → חסימה זמנית**.
- זהו throttle רגעי, לא חסימת חשבון. תואם את הידע הקיים שלנו (403 קשיח תחת קריאות מהירות).

**2. מכסת פעולות חודשית לפי מסלול — "פעולות" (Actions) = היחידה של ActionsBilling.**
- מקור: תוסף WooCommerce (5830000): כל מסלול כולל מספר קבוע של פעולות + קריאות API ללא עלות; **כל קריאת API = פעולה אחת לצורך חיוב**; "יש עלות קטנה נוספת לכל פעולה ו/או קריאת API מעל" המכסה.
- סליקה והפקת מסמך דרך תוסף/אוטומציה נספרות כפעולות רגילות.
- התעריפים המדויקים לא בתיעוד העזרה — בדף מחירון נפרד.
- מסלול חינמי: עד 10 חשבוניות/קבלות בחודש (5835290).
- ב-multi-org: חיובי הפעולות של עסק משולם מתגלגלים לאמצעי התשלום של העסק הראשי (10792013).

**3. חסימת "obligo" — צבירת חוב פעולות + כשל גבייה = המערכת נחסמת.**
- מקור: "הערות מתקדמות על שיטת התמחור" (6247623):
  - חיוב על פעולות עודפות נדחה בד"כ למועד החיוב הבא.
  - **אבל כשעלות הפעולות הנוספות שנצברו עולה על עלות המסלול עצמו — המערכת מבצעת "פריקת חוב": ניסיון גבייה מיידי באמצע החודש.**
  - ציטוט מדויק: **"במקרה שפריקת החוב לא תצלח במשך מספר ניסיונות, המערכת נחסמת לעבודה."**
- זהו כמעט בוודאות המנגנון מאחורי "exceeding the ActionsBilling obligo": ה-obligo (מונח בנקאי לתקרת חשיפת אשראי) = תקרת החוב הצבור על פעולות ש-SUMIT מוכנה לספוג בלי גבייה מיידית — בסדר גודל של עלות המסלול החודשי. חריגה ממנה מפעילה פריקת חוב; כשל בגבייה (כרטיס פג/נדחה, הרשאת בנק חסומה) → חסימת עבודה מלאה.
- **הערה מתודולוגית:** המחרוזת "ActionsBilling obligo" לא מופיעה כלשונה באף מאמר; המיפוי לעיל הוא השחזור המבוסס ביותר מהתיעוד הציבורי, אך לא אישור מילולי.

### מה זה אומר עבורנו (חשבון חסום כרגע)
1. **זו כנראה בעיית חיוב/גבייה, לא בעיית קצב קריאות.** הפחתת קריאות API (כפי שכבר בוצע — ביטול creditguy/billing/load השעתי) מונעת צבירת חוב עתידית אבל לא פותחת חסימה קיימת.
2. **צעדים לשחרור החסימה:** (א) לוודא שאמצעי התשלום הרשום ב-SUMIT תקף (כרטיס בתוקף / הרשאת חיוב פעילה — עמלת כישלון חיוב בהוראת קבע: 50 ₪ + מע"מ); (ב) לפרוע ידנית את יתרת החוב על הפעולות; (ג) לשקול **שדרוג מסלול** — מגדיל מיידית את מכסת הפעולות עד סוף החודש ומזכה על פעולות ששולמו עד תקרת המסלול החדש, כלומר עשוי "לספוג" חלק מהחוב.
3. **לא חשוף בתיעוד הציבורי:** הסף המספרי המדויק של ה-obligo, מספר ניסיונות הגבייה, ותהליך שחרור ידני. נדרשת פנייה לתמיכת SUMIT (התמיכה לא עוזרת בקוד, אבל כן בענייני חשבון/חיוב).

### מפתחות API ואבטחה
- מפתח ציבורי (זיהוי) + פרטי (אימות); הפרטי נחשף רק בהפקה ראשונית (15532017).
- סביבת בדיקות: ארגון נפרד + מסוף בדיקות + כרטיסי בדיקה; **חיבור מסוף בדיקות לחשבון אמיתי מנתק את הסליקה הקיימת ודורש בקשה חדשה בתשלום** (5840939).
- PCI DSS Level 1; מבדקי אבטחה רבעוניים; הרשאת גישה של צוות התמיכה פוקעת אחרי 14 יום (5840795).
- אין מנגנון חתימה (signature) מתועד ל-webhooks יוצאים — לאמת בצד המקבל בדרך אחרת (11577644).

### גיבוי, יצוא ומחיקת נתונים
- SUMIT מגבה שוטף בצד שלה, אבל **בהפסקת תשלום/ביטול אין התחייבות לגישה או לעותקים של המידע** (10340800) — לגבות עצמאית (CSV/אקסל/מבנה אחיד) לפני כל שינוי בהתקשרות. רלוונטי במיוחד כשהחשבון חסום על חוב.
- מחיקת נתונים לפי בקשה או בסיום התקשרות, למעט עותקים להגנה משפטית (10307867); פניות פרטיות/מחיקה: support@sumit.co.il, DPO: dpo@sumit.co.il.
